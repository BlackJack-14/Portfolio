How to Fine-Tune LLMs with LoRA: A Practical Guide for AI/ML Students (2025)

✍️ Rudra Gupta 📅 May 1, 2025 ⏱ 10 min read 🏷 LLM · LoRA · Fine-Tuning · PEFT
📌 Quick Answer — Featured Snippet

LoRA (Low-Rank Adaptation) fine-tuning lets you customize a pre-trained LLM on your own dataset using < 1% of original parameters. The process: (1) load a quantized base model (LLaMA, Mistral, etc.), (2) inject LoRA adapter layers using PEFT, (3) train on your dataset with a small learning rate, (4) merge and export. Free GPU (Google Colab T4) is sufficient for 7B models.

What is LoRA and Why Does It Matter?

Fine-tuning a large language model used to require 80GB+ of GPU VRAM — far beyond reach for students and independent researchers. LoRA changed that entirely.

Introduced in the paper "LoRA: Low-Rank Adaptation of Large Language Models" (Hu et al., 2021), LoRA is a parameter-efficient fine-tuning (PEFT) technique that injects small trainable matrices into a frozen model. Instead of updating billions of weights, you train only a few million — achieving comparable results at a fraction of the computational cost.

This is exactly what I used to build Edu-SLM — a curriculum-aligned LLM fine-tuned on academic content using LoRA on LLaMA 3.1 8B, running on a single GPU.

How LoRA Works: The Core Concept

The Math (Simplified)

In a standard transformer, each weight matrix W is frozen. LoRA adds two small matrices — A and B — whose product approximates the weight update:

# Standard: W is updated (expensive, billions of params) W_new = W + ΔW # LoRA: Only A and B are trained (rank r << d) W_new = W + B × A # A: (r × d_in), B: (d_out × r) # Typical r = 8 or 16 → 10x-50x fewer params than full FT

The rank r controls the expressiveness vs. efficiency tradeoff. For most fine-tuning tasks, r=8 to r=64 works well.

What Gets Adapted?

LoRA is typically applied to the attention weight matrices (Q, K, V, and output projection) in transformer layers. The feed-forward layers are usually left frozen, further reducing trainable parameters.

Step-by-Step: Fine-Tuning LLaMA 3 with LoRA

Step 1: Setup Environment

pip install transformers peft accelerate bitsandbytes datasets trl # For even faster training, install Unsloth: pip install "unsloth[colab-new] @ git+https://github.com/unslothai/unsloth.git"

Step 2: Load Base Model in 4-bit

from transformers import AutoModelForCausalLM, BitsAndBytesConfig import torch bnb_config = BitsAndBytesConfig( load_in_4bit=True, bnb_4bit_quant_type="nf4", bnb_4bit_compute_dtype=torch.float16, ) model = AutoModelForCausalLM.from_pretrained( "meta-llama/Meta-Llama-3.1-8B-Instruct", quantization_config=bnb_config, device_map="auto", )

Step 3: Inject LoRA Adapters

from peft import LoraConfig, get_peft_model lora_config = LoraConfig( r=16, # rank lora_alpha=32, # scaling factor target_modules=["q_proj", "v_proj"], lora_dropout=0.05, bias="none", task_type="CAUSAL_LM", ) model = get_peft_model(model, lora_config) model.print_trainable_parameters() # trainable params: 4,194,304 || all params: 8,034,795,520 # → Only 0.05% of parameters are trained!

Step 4: Prepare Your Dataset

Format your data as instruction-response pairs. For Edu-SLM, I used curriculum-aligned QA pairs extracted from textbooks and structured as chat templates:

dataset_format = { "instruction": "Explain the concept of gradient descent.", "response": "Gradient descent is an optimization algorithm..." }

Step 5: Train with SFTTrainer

from trl import SFTTrainer from transformers import TrainingArguments trainer = SFTTrainer( model=model, train_dataset=dataset, args=TrainingArguments( output_dir="./lora-output", per_device_train_batch_size=2, gradient_accumulation_steps=4, learning_rate=2e-4, num_train_epochs=3, fp16=True, ), ) trainer.train()

LoRA Hyperparameters — What Actually Matters

  • Rank (r): Higher rank = more capacity but more memory. Start with r=8 or r=16.
  • Alpha: Keep alpha = 2× rank as a rule of thumb (e.g., r=16, alpha=32).
  • Target modules: q_proj and v_proj is standard. Add k_proj and o_proj for stronger adaptation.
  • Learning rate: 1e-4 to 3e-4 works well for most tasks. Too high causes instability.
  • Epochs: 1–3 epochs usually sufficient. More can cause overfitting.

Combining LoRA with RAG (My Approach in Edu-SLM)

Fine-tuning alone has limitations — the model can still hallucinate or give outdated answers. In Edu-SLM, I combined LoRA with Retrieval-Augmented Generation (RAG):

  • Textbook content is chunked and embedded into a FAISS vector index
  • At inference time, the query retrieves top-k relevant chunks
  • Retrieved context is injected into the prompt before the fine-tuned model generates a response
  • This gives both domain-specific style (from LoRA) and factual grounding (from RAG)

Frequently Asked Questions

LoRA (Low-Rank Adaptation) is a parameter-efficient fine-tuning technique that injects trainable low-rank matrices into a frozen pre-trained model. Instead of updating all model weights, LoRA only trains a small number of additional parameters, reducing GPU memory requirements by up to 10x while achieving comparable performance to full fine-tuning.

Yes. Google Colab or Kaggle Notebooks offer free T4/P100 GPUs sufficient for LoRA fine-tuning of 7B models in 4-bit quantization. Tools like Unsloth further reduce memory usage, making it possible to fine-tune LLaMA 3 8B on a single free GPU in under 2 hours.

Full fine-tuning updates all model parameters (billions of weights), requiring massive GPU memory and risking catastrophic forgetting. LoRA freezes the original weights and only trains small rank-decomposition matrices (< 1% of total parameters), making it 10–50x more memory-efficient while preserving the base model's general capabilities.

RAG vs Fine-Tuning: Which Should You Choose for Your AI Application in 2025?

✍️ Rudra Gupta 📅 April 15, 2025 ⏱ 12 min read 🏷 RAG · Fine-Tuning · LLM Architecture · AI Design
📌 Quick Answer — Featured Snippet

RAG vs Fine-Tuning at a glance: Use RAG when you need dynamic knowledge access, source citations, or frequent updates. Use fine-tuning when you need the model to behave differently, adopt a new tone/format, or learn domain-specific reasoning. Use both together for production AI applications that need grounded, specialized responses — as in Edu-SLM.

The Core Problem: LLMs Have Knowledge Gaps

Every LLM has a training cutoff and knows nothing about your private documents, your company's internal data, or specialized domain knowledge you've accumulated. To bridge this gap, developers have two primary approaches:

  • Retrieval-Augmented Generation (RAG): Give the model access to external documents at inference time
  • Fine-tuning: Bake new knowledge and behavior directly into the model weights

The question isn't which is "better" — it's which fits your problem. Both have entirely different tradeoffs.

What is RAG? How It Works

The RAG Pipeline

RAG follows a retrieve-then-generate pattern:

  • Indexing: Your documents are chunked and embedded into a vector database (FAISS, Chroma, Pinecone)
  • Retrieval: At query time, the top-k most semantically similar chunks are retrieved
  • Augmentation: Retrieved chunks are injected into the prompt as context
  • Generation: The LLM generates a response grounded in the provided context

I implemented this architecture in TheGeetaWay — using FAISS + Sentence-Transformers to retrieve relevant Bhagavad Gita verses, then feeding them to Llama 3 for context-aware responses. The model never needed fine-tuning because the knowledge was fully contained in the retrieval index.

When RAG Shines

  • Knowledge that changes frequently (news, research papers, product catalogs)
  • Applications requiring source citations and verifiable answers
  • Large, searchable document corpora (> 10k documents)
  • When you want zero GPU training cost
  • When updating knowledge should happen instantly (just add to vector store)

What is Fine-Tuning? How It Works

The Fine-Tuning Paradigm

Fine-tuning modifies the model's weights by training on a task-specific dataset. With LoRA (see my LoRA fine-tuning guide), this process is memory-efficient and fast.

Unlike RAG, the knowledge is baked into the model parameters — no external retrieval step at inference. The model itself becomes the expert.

When Fine-Tuning Shines

  • Teaching the model a new output format (JSON, code, tables)
  • Changing response style or tone (formal, clinical, conversational)
  • Teaching domain-specific reasoning patterns (medical diagnosis, legal interpretation)
  • Improving performance on a narrow, well-defined task
  • When inference speed matters (no retrieval latency)

The Best of Both: RAG + Fine-Tuning Together

In my Edu-SLM project, I combined both — and this hybrid approach is increasingly what production AI systems use:

  • LoRA fine-tuning taught the model academic response style, structured explanation format, and educational reasoning patterns
  • RAG provided real-time retrieval of curriculum-specific content from a FAISS-indexed textbook corpus
  • Result: A model that responds like an educator and answers from verified sources

This combination is sometimes called "fine-tuned RAG" or "grounded fine-tuning" and represents the current state-of-the-art for domain-specific LLM applications.

Decision Framework: Which Should You Choose?

Ask yourself these three questions:

  • "Does my knowledge change frequently?" → Yes: use RAG. No: consider fine-tuning.
  • "Do I need to change how the model behaves?" → Yes: fine-tuning. No: RAG + prompt engineering.
  • "Do I have GPU resources?" → No: RAG only. Yes: evaluate fine-tuning.

Frequently Asked Questions

Use RAG when your application needs access to dynamic, frequently updated information; when you need to cite sources; when you have a large, searchable knowledge base; or when you want to avoid expensive retraining. RAG is faster to deploy and easier to update — you just add documents to the vector store.

Neither is universally better — they solve different problems. RAG excels at providing up-to-date, retrievable knowledge. Fine-tuning excels at changing the model's behavior, style, and reasoning patterns. For most production AI applications, combining both delivers the best results.

RAG has negligible upfront cost — you just build a vector database. Fine-tuning requires GPU compute (1–10 hours for LoRA on a 7B model) but is a one-time cost. At inference time, RAG adds latency from retrieval. Fine-tuned models are self-contained but require retraining when knowledge updates.
